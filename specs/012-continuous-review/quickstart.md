## 動作確認（ローカル）

### 準備

```bash
supabase start                # ローカル Supabase（起動していない場合のみ）
supabase db reset             # 必要なら（マスタ再投入含む）
pnpm tsx scripts/sync-municipality-master.ts   # municipality_master が空の場合
pnpm dev
```

ブラウザで `http://127.0.0.1:3000` を開く。

> [!NOTE]
> **ログインについて**
> ローカル環境ではメール確認（Email Confirmation）が無効化されています。アカウントが存在しない、あるいはログインできない場合は、新規アカウント作成画面 (`/signup`) から適当なメールアドレスとパスワード（6文字以上）を入力して登録してください。確認メールの受信を待つことなく即座にアカウントが作成され、ログインできます。

### 事前準備: 期日超過の復習項目を21件以上作る

Studio（http://127.0.0.1:54323）の SQL Editor、または `psql` で対象ユーザーの `srs_records` を直接操作し、`status='reviewing'` かつ `due_date` を過去日時にした行を21件以上用意する。あるいは、実際に複数の市区町村クイズをプレイして誤答を積み重ね、`due_date` が過去になるまで日付を進める（手間が大きいため SQL 直操作を推奨）。

### 動作確認手順

#### 1. 続行アクションの表示と連続プレイ（US1 / FR-001〜004）
1. 期日超過の復習項目を21件以上持つ状態でログインし、`/quiz/review` を開く。
2. 1バッチ目（20件）をプレイし終える。
3. 結果画面に「続けて復習する」アクションが表示されることを確認する。
4. それを選択し、ダッシュボードを経由せず次のバッチのプレイが直接始まることを確認する。

#### 2. バッチごとの成績非合算（FR-005）
1. 上記1の1バッチ目で意図的に数問間違え、正答率・苦手一覧を記録しておく。
2. 2バッチ目を（全問正解などにして）終える。
3. 2バッチ目の結果画面に表示される正答率・苦手一覧が、2バッチ目単体の成績のみであり、1バッチ目の内容と合算されていないことを確認する。

#### 3. 残り件数の表示（US2 / FR-009）
1. 期日超過の復習項目が既知の件数（例: 13件）になるよう調整し、バッチを終える。
2. 結果画面の続行アクションに残数（13件）が表示されることを確認する。

#### 4. 残り0件時のフォールバック（FR-006 / FR-008 / Edge Cases）
1. 期日超過の復習項目がちょうど1バッチ分（20件以下）しかない状態でプレイを終える。
2. 続行アクションが表示されず、既存の完了体験（ダッシュボードへ／おすすめクイズを試す）のみが表示されることを確認する。
3. （続行後に0件になるケース）21件保持した状態で1バッチ目を終えて「続ける」を押し、2バッチ目が期日超過分の残り全て（21件未満）で完了した場合、続行アクションが表示されず「今日の復習はありません」の完了状態が表示されることを確認する。

#### 5. 残数取得失敗時のフォールバック（FR-007）
1. ブラウザの DevTools でネットワークをオフラインにするなどして `getDueReviewSummary()` の取得を失敗させた状態でバッチを終える。
2. 「続けて復習する」ボタン自体は（件数表示なしで）引き続き表示され、選択すると次のバッチが直接始まることを確認する（自動テストでの代替確認は困難なため、目視確認が必須）。

#### 6. リファクタの回帰確認（due 判定共通化）
1. `getDueReviewSummaryData` を使うダッシュボードの「今日の復習」表示件数と、`/quiz/review` で実際にプレイできる件数が一致することを確認する（B013 で修正済みの整合性が本リファクタ後も崩れていないことの確認）。
