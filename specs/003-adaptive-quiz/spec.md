# Feature Specification: おすすめクイズ（適切クイズ推薦）

**Feature Branch**: `003-adaptive-quiz`
**Created**: 2026-05-28
**Status**: Draft
**Input**: User description: "今の自分に適切なquizを実施できるようにしたい"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ワンタップで「自分に適したクイズ」を開始 (Priority: P1)

ユーザーがダッシュボードやクイズ選択画面から「おすすめクイズを始める」ボタンを1タップするだけで、自分の学習状態に合わせて出題内容（対象市区町村・難易度・地方・問題数）が自動的に組み立てられた1セッションを開始できる。設定画面で都度パラメータを選び直す必要がない。

**Why this priority**: 「設定が多くて何を選べばいいか分からない」「同じ条件ばかりプレイしている」というユーザーの摩擦を解消し、継続学習の入り口を最も短くする。本フィーチャーの根幹であり、これ単体で MVP として機能する。

**Independent Test**: ダッシュボードまたはクイズトップに表示された「おすすめクイズ」ボタンをタップし、設定確認なしまたは最小限の確認のみでクイズが開始され、最後まで完走できれば独立して機能する。

**Acceptance Scenarios**:

1. **Given** ユーザーがダッシュボードを開くと、**When** 「おすすめクイズ」ボタンをタップすると、**Then** 学習履歴に基づき自動選定された問題セット（10〜20問）が即座に開始される。
2. **Given** クイズトップ画面で、**When** 「おすすめクイズ」エントリーをタップすると、**Then** ダッシュボードと同じ推薦エンジンで生成されたセッションが開始される。
3. **Given** クイズ履歴が一切ないユーザーが、**When** 「おすすめクイズ」をタップすると、**Then** 全国全モード・☆入門難易度・10問のオンボーディング向けセッションが開始される（コールドスタート対応）。
4. **Given** ユーザーが推薦されたクイズを完走すると、**When** 結果画面が表示されると、**Then** 「もう一度おすすめでプレイ」ボタンが表示され、最新の学習状態を反映した新しい推薦セッションへ遷移できる。

---

### User Story 1a - 多軸定義による「適切なレベル」の自動構成 (Priority: P1)

US1 と一体で動作する内部要件。「適切なレベル」を 4 軸（正答率・カバレッジ・成長促進・多様性）で定義し、各軸を同時に満たす推薦セットを構成する。詳細は [§ 適切なレベルの定義（多軸モデル）](#適切なレベルの定義多軸モデル) を参照。

**Why this priority**: P1 の「ワンタップ開始」を実現する内部ロジックの根幹。単に苦手だけを並べる、単に未経験だけを並べる、単に難易度だけ調整する、ではなく、4 軸を組み合わせて初めて「ちょうどいい」と感じられるセッションになる。

**Independent Test**: 同一ユーザーの異なる学習状態スナップショット（履歴ゼロ / 履歴少 / 中堅 / 上級 / 全制覇）を入力に、推薦エンジンが各軸の制約を満たす出力を返せれば独立して検証できる。

**Acceptance Scenarios**:

1. **Given** 適切ゾーン（移動平均 60〜80% のセル集合）が ☆入門の複数セルで構成され、その最高バケット内で直近 5 セッション平均が 80% 以上のユーザーが、**When** 「おすすめクイズ」をタップすると、**Then** ☆☆中級の隣接バケットを適切ゾーンに追加した推薦セットが返される（成長軸・隣接単方向）。
2. **Given** ☆☆中級で安定しているユーザーが ☆☆☆☆達人へのスキップを期待しても、**When** 「おすすめクイズ」をタップすると、**Then** 適切ゾーンに追加されるのは ☆☆☆上級までであり、☆☆☆☆達人は含まれない（昇格スキップ禁止）。
3. **Given** 推薦エンジンが未プレイのセル（例: 中国地方 × ☆☆ × モード C）を保有するユーザーが、**When** 「おすすめクイズ」をタップすると、**Then** 推薦セッションの 20〜30% がそのような未プレイ or 低制覇セルで構成される（探索軸）。
4. **Given** 推薦セル内に未経験の市区町村があるユーザーが、**When** 「おすすめクイズ」をタップすると、**Then** 推薦市区町村の 20% 以上が未経験市区町村で構成される（カバレッジ軸）。
5. **Given** 直近セッションの正答率が 30% を割ったユーザーが、**When** 「おすすめクイズ」をタップすると、**Then** 適切ゾーン内の復習を強化した推薦セットが返され、難易度の隣接拡張は行われない（後退抑制）。
6. **Given** ☆☆☆☆達人 × モード A で適切ゾーンが収束し 80% 張り付きのユーザーが、**When** 「おすすめクイズ」をタップすると、**Then** 難易度はそれ以上拡張できないため、別地方への広域化 or より挑戦的なモードへの切り替え提案が含まれる（達人天井ケース）。

### User Story 2 - 推薦内容の根拠を確認 (Priority: P2)

ユーザーがおすすめクイズを開始する前または開始直後に、「なぜこの内容が選ばれたか」を短文で確認できる。例: 「東北地方で苦手だった市区町村を中心に出題します」「最近触れていない☆☆中級を復習します」。これによりブラックボックスへの不信感を払拭し、推薦結果に納得して取り組める。

**Why this priority**: P1 単独でも機能はするが、「なぜこれが出題されたか分からない」という不信感は継続率を下げる。短い説明文を1行追加するだけで信頼性を大きく高められる。

**Independent Test**: おすすめクイズ開始時に推薦理由のテキストが表示され、その内容が出題セットと整合していれば独立して機能する。

**Acceptance Scenarios**:

1. **Given** 推薦エンジンが苦手市区町村を中心に選定したとき、**When** ユーザーが開始画面を見ると、**Then** 「あなたの苦手 N 件を中心に出題します」のような根拠文が1〜2行で表示される。
2. **Given** 推薦エンジンが復習タイミングに基づき選定したとき、**When** 開始画面を見ると、**Then** 「最後にプレイから N 日経過した問題を復習します」と表示される。
3. **Given** 推薦エンジンが新規探索を含めたとき、**When** 開始画面を見ると、**Then** 「未挑戦の市区町村 N 件を含みます」と表示される。
4. **Given** コールドスタート（履歴ゼロ）ユーザーのとき、**When** 開始画面を見ると、**Then** 「初めての方向けに ☆入門 を全国から出題します」と表示される。
5. **Given** 成長促進軸が発火して難易度ステップアップしたとき、**When** 開始画面を見ると、**Then** 「☆入門の正答率が安定したので、☆☆中級に挑戦しましょう」のような昇格根拠文が表示される。
6. **Given** 成長促進軸が発火してモード変更したとき、**When** 開始画面を見ると、**Then** 「モード B（4択）の正答率が安定したので、モード A（地図）に挑戦しましょう」のような根拠文が表示される。
7. **Given** 後退抑制が発火したとき、**When** 開始画面を見ると、**Then** 「少しペースを落として、現在のレベルを確実にしましょう」のようなトーンの根拠文が表示される。

### User Story 3 - 推薦内容を開始前に微調整 (Priority: P3)

ユーザーが推薦されたクイズの内容を、開始ボタンを押す前に最小限のパラメータ（モード、問題数、地方の追加除外）で上書き調整できる。「気分的に今日はモード C をやりたい」「東北は除外して関東に集中したい」といった希望に応える。

**Why this priority**: 全自動は便利だが、ユーザーが「今日は別の気分」というケースに対応できないと使い勝手が落ちる。最小限のオーバーライドUIで柔軟性を確保する。

**Independent Test**: 推薦画面でモードや地方を変更してから開始ボタンを押すと、上書きされたパラメータでクイズが開始されれば独立して機能する。

**Acceptance Scenarios**:

1. **Given** 推薦内容の確認画面で、**When** ユーザーがモードを A→C に変更すると、**Then** 同じ対象市区町村プールでモード C のクイズが開始される。
2. **Given** 推薦内容の確認画面で、**When** ユーザーが問題数を 10→20 に変更すると、**Then** 推薦プールから 20 問が生成される。プール不足時は推薦範囲外の市区町村で補充され、その旨が注記される。
3. **Given** 推薦内容の確認画面で、**When** ユーザーが「東北地方を除外」を選ぶと、**Then** 東北以外の推薦市区町村でセッションが構成される。
4. **Given** ユーザーがオーバーライドせず「そのまま開始」を選ぶと、**When** クイズが始まると、**Then** 推薦エンジンの提示内容そのままで進行する。

### User Story 4 - 結果に基づく次回推薦への反映 (Priority: P4)

ユーザーがおすすめクイズを完了すると、そのセッションの正誤結果が即座に推薦エンジンに反映され、次回の「おすすめクイズ」では最新の弱点・復習状況に応じた問題が選ばれる。

**Why this priority**: 推薦内容が更新されないと「いつも同じ問題が出る」という感覚を生む。継続的な学習サイクルのために、結果反映は重要だが、P1-P3 が動けば本質的価値は提供されるので P4 とする。

**Independent Test**: おすすめクイズを完了し、その回で間違えた市区町村が、直後の次回推薦セッションの対象プールに含まれていれば独立して機能する。

**Acceptance Scenarios**:

1. **Given** ユーザーがおすすめクイズで特定の市区町村を不正解にすると、**When** 直後に再度おすすめクイズを開始すると、**Then** その市区町村は次回推薦の候補プールに優先的に含まれる。
2. **Given** ユーザーがおすすめクイズで全問正解した市区町村があると、**When** 次回推薦を開始すると、**Then** 直近正解した市区町村は復習対象から外され、別の弱点や新規市区町村が選ばれる。
3. **Given** ユーザーが連続で同一モードのおすすめクイズをプレイすると、**When** 学習状況が変化するに伴い、**Then** 出題プールが少しずつ変化し、同じ問題セットが繰り返されない。

---

### Edge Cases

- **コールドスタート（履歴ゼロ）**: 全モード・☆入門・全国・10問のオンボーディング向けセッションを既定値として返す。推薦根拠文は「初めての方向け」とする。
- **データ過小（履歴 1〜10 件）**: 苦手データが十分に集まっていないため、☆入門〜☆☆中級の混合 + 既出題エリアの近隣を中心とした「学習継続向け」セッションを返す。
- **全問正解済みユーザー（市区町村制覇後）**: 全体正答率の低い難易度バケットを再復習対象として推薦する。それも 100% のときは「全市区町村制覇済み！ ランダム挑戦モード」を提示する。
- **推薦プールが要求問題数に満たない**: プール不足分は推薦範囲外の市区町村からランダム補充し、確認画面と結果画面にその旨を注記する。
- **直近プレイから時間が空いた場合（例: 1 週間以上ぶり）**: 「お帰りなさい」メッセージと共に、まだ覚えているか確認するための既習復習中心のセッションを提示する。
- **同日内で複数回おすすめクイズを完走**: 直前セッションの結果を反映して次回出題内容を変える。ただし「直前と全く同じ問題」になることは避ける（少なくとも 50% は入れ替える）。
- **モード A/D の地図プールが少なすぎる地方**: 該当地方を除外せざるを得ない旨を根拠文に明示する。
- **推薦エンジンが何も提案できない場合**: フォールバックとして既存のランダム出題（モード・難易度・地方ランダム）を使用する。
- **成長促進軸が頻繁に発火しすぎる**: 直近 3 セッション平均で 80% に到達して昇格した直後、新レベルで正答率が落ちて元のレベルに戻る、を短期で繰り返す状況を避ける。具体的には昇格直後の 1 セッションは後退抑制を適用せず、2 セッション目以降で後退条件（< 30%）を再評価する。
- **カバレッジ軸が満たせない（全市区町村経験済み）**: 未経験プールが空になったユーザーには、「最終出題から 30 日以上経過した既習プール」を代替の探索源とする。それも空ならカバレッジ下限を 0% に緩める。
- **正答率軸と成長促進軸が衝突**: 直近正答率は 80% 超だが、次レベル（より難しい難易度・モード）に進むと想定正答率が 60% を割る場合、半数を現レベル復習・半数を次レベル挑戦の混合セッションで橋渡しする。
- **都道府県クイズとの関係**: 本フィーチャーの対象は市区町村クイズのみ。都道府県クイズの結果は推薦エンジンに使用しない（B003 で別途検討）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはダッシュボード（`/`）の **`MilestoneBanner` の直下、`SummaryCards` の上** に「おすすめクイズを始める」ヒーローカードを表示しなければならない。カードには「タイトル / 推薦根拠文 1 行 / 構成サマリー（モード・難易度・地方・問題数）/ 主 CTA ボタン」を含める。CTA タップ時は **ボトムシート（モーダル）** で推薦内容の確認 UI を表示し、ページ遷移は発生させない。シート内に推薦内容・根拠文・上書きオプション（US3）を集約し、「そのまま開始」ボタンをシート最下部に sticky 配置する。デフォルト動作で CTA → シート表示 → 「そのまま開始」 → 第 1 問の 2 タップで開始できるようにする。
- **FR-002**: システムは市区町村クイズトップ画面（`/quiz/municipality`）の **モード A/B/C/D カード群の上**（最上段）に「おすすめクイズ」ヒーローカードを表示しなければならない。デザイン言語はダッシュボードのカードと共通とし、タップ時は同じボトムシートが開く。
- **FR-003**: システムは推薦エンジンを通じて、各ユーザーごとに「対象市区町村プール・推奨モード・推奨難易度・推奨地方・推奨問題数」を含む推薦セットを生成しなければならない。推奨問題数（10/20/30）は **ユーザーの直近 10 セッションの問題数の最頻値（同率なら大きい方）** を採用する。コールドスタート（履歴ゼロ）は 10 問とする。ユーザーが確認画面で上書きすれば次回以降の最頻値計算にもその選択が反映される。
- **FR-004**: 推薦エンジンは少なくとも次の入力シグナルを使用しなければならない:
  - 苦手度（市区町村ごとの誤答率）
  - 復習タイミング（最終出題日からの経過日数）
  - 既出題実績（一度も出題されていない市区町村の有無）
  - 直近プレイ済みかどうか（同セッションでの再出題防止）
  - 現在の難易度・モードでの直近 N セッション平均正答率（成長促進判定用）
- **FR-005**: 推薦エンジンは「適切なレベル」を **「適切ゾーン（Fit Zone）」** という概念で捉え、以下の 4 軸を同時に満たすよう推薦セットを構成しなければならない（詳細は [§ 適切なレベルの定義（多軸モデル）](#適切なレベルの定義多軸モデル) を参照）:
  1. **適合軸（旧 正答率軸）** — セル別正答率移動平均が 60〜80% に収まるセル集合を「適切ゾーン」とし、推薦の中核プールとして使用する。
  2. **探索軸（旧 多様性軸）** — 未プレイのセル、または cell-level 制覇率（セル内市区町村のうち 1 回以上正解した割合）が低いセルを推薦に組み込む。
  3. **成長軸（旧 成長促進軸）** — 適切ゾーンが現在の最高難易度バケットに収束して上限に張り付いたとき、難易度を **隣接バケット 1 段上** に拡張する。
  4. **カバレッジ軸** — 推薦に含まれるセル内で、ユーザー未経験の **市区町村** を一定割合確保する（cell 内 municipality-level の探索）。
- **FR-006**: 正答率軸 — 推薦エンジンは「想定正答率」を **(難易度バケット × 地方 × モード) のセル単位** で算出しなければならない。算出方法:
  - セッション単位の正答率（正答数 ÷ 出題数）を計算し、各セッションを「セッション内で最多を占める (難易度, 地方, モード) セル」に紐づける（同一セル内の出題が 50% 以上のときに限り紐づけ、満たさないセッションは「混合」として除外）。
  - 各セルごとに **直近 5 セッションの移動平均** を算出して、そのセルの想定正答率とする。
  - 推薦エンジンは、想定正答率が 60〜80% に収まるセルを優先的に選び、低い（< 60%）セルは難易度を下げる方向、高い（> 80%）セルは次の難易度や別モードへ昇格する方向（成長促進軸へ）に推薦を組み立てる。
  - セッション数 < 3 のセルは隣接セルから補完する: 同モード × 同難易度（全地方平均）→ 同モード × 全難易度（全地方平均）→ 全モード × 全難易度の順にバックオフ。
  - 完全に未プレイのセルは `municipality_master.difficulty` のバケットに対するクラウド平均（全ユーザー集計）で初期化する。
- **FR-006a**: 探索軸 — 推薦エンジンは、適切ゾーンに含まれないセルのうち「未プレイのセル」または「cell-level 制覇率が下位 25% に位置するセル」を、推薦に最低限の割合（標準 20〜30%）で組み込まなければならない。これにより「まだ全然触れていないところ」や「苦手なまま放置されているセル」へ自然に手が伸びる。
- **FR-006b**: 成長軸 — 推薦エンジンは、ユーザーの適切ゾーンが「現在プレイしている最高難易度バケット」内のみで構成され、その最高バケットの直近 5 セッション平均正答率が 80% 以上に張り付いている場合、難易度を **1 段上の隣接バケット** に拡張しなければならない。難易度の順序は厳格に **☆入門 → ☆☆中級 → ☆☆☆上級 → ☆☆☆☆達人** とし、スキップ（例: ☆ → ☆☆☆）は禁止する。
  - 達人（最上位）に到達して張り付いた場合は難易度の拡張ができないため、副次的に **モード変更**（同一難易度内でより挑戦感のあるモードへの切り替え）または **別地方への広域化** で挑戦感を維持する。
  - モード変更は本来、難易度の昇格手段ではなく、同一難易度内の挑戦バリエーションとして扱う。
- **FR-006c**: カバレッジ軸 — 推薦に含まれる各セル内で、ユーザーがまだ 1 回も正解していない市区町村を、推薦市区町村の最低 20%（標準 30%）含めなければならない。cell 内の未経験市区町村が枯渇している場合は、最終出題から 30 日以上経過した既習市区町村で代替する。
- **FR-006d**: 後退抑制 — 直近セッションが極端な不正解率（< 30%）だった場合、成長軸の難易度拡張を 1 セッション分保留し、現在の適切ゾーン内での復習を強化しなければならない。フラストレーション蓄積を回避するため。
- **FR-007**: 推薦内容の確認画面では、推薦根拠を1〜2行の自然言語で表示しなければならない。例: 「あなたの苦手 N 件を中心に出題します」「☆☆中級の復習を中心に出題します」。
- **FR-008**: 推薦内容の確認画面では、ユーザーがモード（A/B/C/D）・問題数（10/20/30）・除外地方 を上書きして開始できなければならない。難易度と対象市区町村プールはユーザーが直接編集できない（推薦エンジン管轄）。
- **FR-009**: クイズ履歴がないユーザー（コールドスタート）には、全モード・☆入門・全国・10問のオンボーディングセッションを返さなければならない。
- **FR-010**: 履歴が極端に少ないユーザー（例: 10 問未満）には、☆入門〜☆☆中級の混合 + 既出題エリア近隣中心のセッションを返さなければならない。
- **FR-011**: クイズ完了後の結果画面（既存の `/quiz/municipality/[mode]` の `phase === 'result'`）の **既存「もう一度」「設定に戻る」ボタン群の上に主 CTA として** 「もう一度おすすめでプレイ」ボタンを追加しなければならない。タップ時は再度ボトムシートが開き、最新の学習状態を反映した次の推薦セッションを表示する。
- **FR-012**: 推薦の対象データは市区町村クイズの結果（`municipality_quiz_results`）のみとする。都道府県クイズの結果は対象外。
- **FR-013**: 推薦プールが要求問題数に満たない場合、不足分を推薦範囲外の市区町村からランダム補充し、その旨を確認画面と結果画面に注記しなければならない。
- **FR-014**: 推薦エンジンは CTA タップごとに毎回フル再計算しなければならない（キャッシュなし）。同日内の連続プレイ時に推薦結果が少なくとも 50% 入れ替わるよう、Recommendation History Cache に保持された直前セッションの対象市区町村集合を参照し、推薦エンジン側で意図的に多様性を担保する（同一セットの連発防止）。
- **FR-015**: 全市区町村正解経験済みユーザーには、全体正答率の低い難易度バケットを再復習対象とする推薦を返さなければならない。全難易度で 100% のときはランダム挑戦モードへフォールバックする。
- **FR-016**: 推薦エンジンが推薦不能なエッジケースに陥った場合、既存のランダム出題（モード・難易度・地方ランダム）にフォールバックしなければならない。
- **FR-017**: 推薦セッションは既存のクイズ実行画面（`/quiz/municipality/[mode]`）を再利用し、推薦パラメータを URL クエリまたは内部ステートで引き渡して開始される。新規のクイズ実行 UI は作らない。
- **FR-018**: 推薦エンジンは新テーブルを追加せず、既存の `municipality_quiz_results` および `municipality_master` のクエリ集計で実装されなければならない。状態の永続化が必要な場合はクライアント側ストレージ（直前推薦内容のキャッシュ等）で対応する。

### 適切なレベルの定義（多軸モデル）

「自分に適切なクイズ」を以下の 4 軸で同時に評価する。推薦エンジンは各軸の制約を満たす推薦セットを生成する。

「適切なレベル」を **適切ゾーン（Fit Zone）** = 「セル別正答率移動平均が 60〜80% に収まる (難易度 × 地方 × モード) セル集合」として定義する。推薦エンジンは適切ゾーンを軸に推薦を組み、外側 3 軸で補完する。

| 軸 | 何を測るか | しきい値・目標 | 失敗時の挙動 |
|---|---|---|---|
| **① 適合軸（Fit Zone）** | (難易度 × 地方 × モード) セルごとの直近 5 セッション正答率の移動平均 | 60% ≤ 想定正答率 ≤ 80% のセル集合を「適切ゾーン」とする | データ不足セルは隣接セル → クラウド平均でバックオフして判定する |
| **② 探索軸** | 未プレイ or cell-level 制覇率下位 25% のセル | 標準 20〜30% を推薦に含める | 該当セルがない（全セル既プレイ + 高制覇）場合は最終出題 30 日以上経過セルで代替 |
| **③ 成長軸** | 適切ゾーンが最高難易度バケットに収束 + 直近 5 セッション平均 ≥ 80% | 隣接 1 段上の難易度バケットへ拡張（☆→☆☆→☆☆☆→☆☆☆☆、スキップ禁止） | 後退抑制（FR-006d）: 直近セッション < 30% なら 1 セッション保留。達人最上位はモード変更／別地方広域化で代替 |
| **④ カバレッジ軸** | 推薦セル内のユーザー未経験市区町村の含有率 | 最低 20%（標準 30%） | cell 内未経験プール枯渇時は最終出題 30 日以上経過の既習市区町村で代替 |

**軸間の優先順位（衝突時のルール）**:

1. 後退抑制（FR-006d）が最優先。直近セッションが極端に低正答率なら、他軸より先に難易度拡張を据え置く。
2. 適合軸 ≥ 成長軸 ≥ 探索軸 ≥ カバレッジ軸 の順で制約を解く。例: 適切ゾーンを抽出 → 成長軸で隣接拡張判定 → 探索軸で外側セル取り込み → カバレッジ軸で市区町村レベル未経験を確保。
3. 全軸を同時に満たす推薦セットが構築できないときは、低優先度の軸（カバレッジ → 探索）から制約を緩めて妥協する。

**昇格パスの例（成長軸）**:

- ☆入門 × モード B（4択）で 80% 安定し、適切ゾーンが ☆ のみに収束 → ☆☆中級 × モード B を適切ゾーンに追加（隣接単方向）。
- ☆☆中級 × モード A（地図）で安定 → ☆☆☆上級 × モード A を追加。スキップ（☆ → ☆☆☆ など）は禁止。
- ☆☆☆☆達人 × モード A で 80% 安定 → 難易度は拡張できないため、副次的に別地方への広域化 or より挑戦的なモード（地図系・逆引き系）に切り替える。

### Key Entities *(include if feature involves data)*

- **推薦セット (Recommendation Set)**: ユーザー単位で生成される派生データ。属性: 対象市区町村コードのリスト、推奨モード（A/B/C/D）、推奨難易度バケット集合、推奨地方集合、推奨問題数、推薦根拠文、生成タイムスタンプ。DB 永続化はせず、リクエスト時に算出する。
- **学習状態スナップショット (Learner State)**: 推薦エンジンが内部で使用する派生指標。属性: 市区町村別誤答率、最終出題日、出題回数、ユーザー全体正答率、難易度別正答率、未出題市区町村数、セル別正答率移動平均（後述）、セル別制覇率（後述）。`municipality_quiz_results` と `municipality_master` のクエリで導出する。
- **セル別正答率移動平均 (Cell Accuracy Moving Average)**: (難易度バケット × 地方 × モード) を組み合わせたセル単位で、ユーザーの直近 5 セッション正答率の単純移動平均を保持する派生指標。セッションのセル所属は「セッション内で 50% 以上を占める (難易度, 地方, モード) タプル」で判定し、満たさないセッションは集計から除外する。データ不足セル（過去セッション数 < 3）は隣接セル → クラウド平均でバックオフする。適合軸・成長軸の判定に使用する。
- **セル別制覇率 (Cell Coverage Rate)**: (難易度 × 地方 × モード) セルごとに、セル内市区町村のうちユーザーが 1 回以上正解した割合を保持する派生指標。探索軸（cell-level 下位 25% セル特定）とカバレッジ軸（cell 内未経験市区町村特定）の判定に使用する。
- **適切ゾーン (Fit Zone)**: 適合軸の判定結果として、想定正答率が 60〜80% に収まる (難易度 × 地方 × モード) セル集合。推薦の中核プール。リクエストごとに再計算され、DB 永続化はしない。
- **推薦履歴キャッシュ (Recommendation History Cache)**: 同日連発時の重複防止用に、直前セッションの対象市区町村コード集合を保持する。クライアント側ストレージ（localStorage）で実装し、24 時間で expire する。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがダッシュボードを開いてから「おすすめクイズ」ボタンをタップし、最初の問題が表示されるまでの操作回数が 2 タップ以下（CTA → 開始確認 → 第1問）に収まる。
- **SC-002**: おすすめクイズの 1 セッション完了時の正答率が 60〜80% の範囲に収まる割合が全セッション中 70% 以上である（正答率軸が機能していることの定量検証）。
- **SC-003**: 推薦内容の生成（CTA タップから確認画面表示まで）が 1.5 秒未満で完了する。
- **SC-004**: 履歴ゼロ・履歴少（10問未満）・履歴中（100問以上）・履歴大（1000問以上）の全セグメントで、「おすすめクイズ」ボタンタップ後に何らかのクイズが開始される（エラー画面や空状態にならない）。
- **SC-005**: 同一ユーザーが同日内に「おすすめクイズ」を 2 回連続で開始したとき、対象市区町村集合の重複が 50% 以下になる。
- **SC-006**: フィーチャー導入後、「おすすめクイズ」経由のクイズ完走率（開始→最終問題到達）が、手動設定クイズの完走率と同等以上（85% 以上）。
- **SC-007**: カバレッジ軸 — 全市区町村経験済みでないユーザーの推薦セッションで、未経験プールから出題される割合の平均が 20% 以上である。
- **SC-008**: 成長促進軸 — 「直近 3 セッション平均正答率 80% 以上」の状態を 5 セッション以上維持したユーザーは、その後の推薦セッションで難易度ステップアップまたはモード変更が必ず適用されている。
- **SC-009**: 多様性軸 — 同一ユーザーの連続 3 セッション全てで「モード × 主要難易度 × 主要地方」が完全一致するケースが、全セッション中 5% 以下に抑えられる。

## Clarifications

### Session 2026-05-28

- Q: 「想定正答率」（FR-006）はどう算出するか？ → A: (難易度バケット × 地方 × モード) のセル単位で、セッション正答率の **直近 5 セッション移動平均** を用いる。セッションのセル所属は「セッション内で 50% 以上を占める (難易度, 地方, モード) タプル」で判定し、データ不足セル（< 3 セッション）は同モード × 同難易度 → 同モード → 全モードの順でバックオフ、完全未プレイセルは難易度バケットのクラウド平均で初期化する。
- Q: 「主要難易度・主要モード・主要地方」をどう特定するか？ → A: 単一の主要 X を特定するのではなく、**正答率 60〜80% に収まるセル集合 = 「適切ゾーン（Fit Zone）」** を「現在のレベル」として扱う。多様性軸は「未プレイのセル or cell-level 制覇率が低いセル」を取り込む探索軸として再定義する。難易度の昇格は **☆→☆☆→☆☆☆→☆☆☆☆ の順を崩さない隣接単方向のみ** とし、スキップは禁止。モード変更は同一難易度内の挑戦バリエーションとして扱い、達人（最上位）天井ケースの補助手段としてのみ昇格手段に転用する。
- Q: 「おすすめクイズを始める」CTA タップ後の UX フローは？ → A: 確認画面 1 ステップを挟み、推薦内容・根拠・上書きオプションを 1 画面に集約する。「そのまま開始」を最も目立つボタンとして配置し、デフォルトで CTA → 確認画面 → 第 1 問の 2 タップで開始できるようにする。即時開始（タップ即第 1 問）は採用しない。
- Q: 推薦問題数（10/20/30）の自動選定基準は？ → A: ユーザーの **直近 10 セッションの問題数の最頻値**（同率時は大きい方）を採用する。コールドスタートは 10 問。ユーザーが確認画面で上書きすればその選択が次回以降の最頻値計算にも反映され、自然に習熟度や好みに追従する。
- Q: 推薦の再計算タイミングとキャッシュ戦略は？ → A: CTA タップごとに毎回フル再計算する（キャッシュなし）。TanStack Query は staleTime: 0。FR-014（同日 50% 入れ替え）はキャッシュ無効化ではなく、推薦エンジン内のランダム化 + Recommendation History Cache（直前セッションの対象市区町村集合）参照で実現する。
- Q: UI 配置と確認画面の形態は？ → A: ダッシュボードでは **MilestoneBanner と SummaryCards の間にヒーローカード**、クイズトップでは **モード A/B/C/D カード群の最上段にヒーローカード** を配置。確認画面は **ボトムシート（モーダル）** とし、専用ページは使わない（ページ遷移を避け体感速度を優先）。結果画面では「もう一度おすすめでプレイ」を既存ボタン群の上に主 CTA として配置。詳細は [§ UI 配置と見せ方](#ui-配置と見せ方) を参照。

## Assumptions

- 対象データは市区町村クイズの結果（`municipality_quiz_results`）のみ。都道府県クイズは現時点で結果が DB 保存されていないため対象外とする（B003 で都道府県クイズ強化が予定されている）。
- 既存テーブル（`municipality_quiz_results`、`municipality_master`）のスキーマ変更は不要。新テーブルも追加しない（B002 の Leitner SRS は別 spec として分離）。
- 既存の `getMunicipalityWeakness()`（市区町村別誤答率取得）および `weightedSample()`（重み付きサンプリング）ロジックを推薦エンジンの内部で再利用または拡張する。
- 既存のクイズ実行画面（`/quiz/municipality/[mode]`）を再利用し、推薦エンジンはパラメータ生成のみを担う。新規のクイズ実行 UI は作らない。
- 認証済みユーザー向けの機能とする（未ログインユーザーは推薦に必要な履歴を持たないため）。
- 推薦エンジンの最初の実装は heuristic ベース（既存の苦手優先 + 経過日数 + 難易度調整の組み合わせ）とし、機械学習や統計モデルは導入しない。
- B002 (Leitner spaced repetition) は別 spec で実装する。本 spec の「復習タイミング」は単純な「最終出題日からの経過日数」のみを使用し、Leitner Box の概念は持ち込まない。両者は将来統合可能だが、本 spec のスコープ外。
- 推薦内容の根拠文はテンプレート方式（事前定義された数パターンの組み合わせ）で生成し、LLM 等は使用しない。テンプレートは少なくとも次のカテゴリをカバーする: 苦手中心 / 復習タイミング / 新規探索 / 難易度ステップアップ / モード変更 / 後退抑制 / 混合（橋渡し）/ コールドスタート。
- 多軸モデルの軸間優先順位（適合軸 ≥ 成長軸 ≥ 探索軸 ≥ カバレッジ軸、後退抑制が最優先）は固定とする。ユーザー設定での重み変更は本 spec のスコープ外。
- 「現在のレベル」は単一の主要難易度・主要モード・主要地方ではなく、**適切ゾーン（Fit Zone）= 直近 5 セッション正答率移動平均が 60〜80% に収まるセル集合** として表現する。
- 難易度の昇格は **☆入門 → ☆☆中級 → ☆☆☆上級 → ☆☆☆☆達人** の隣接単方向のみとし、スキップ（例: ☆ → ☆☆☆）は禁止する。
- モード変更は本来は同一難易度内の挑戦バリエーションであり、難易度昇格の代替手段ではない。例外として、達人（最上位）で適切ゾーンが頭打ちになったときのみ、補助的にモード変更を成長手段として使う。
- モバイルファースト UI で実装し、375px 幅でも推薦確認画面が縦スクロールのみで完結する。

## Out of Scope

- Leitner Box / SRS の状態管理（B002 として別 spec）。
- 都道府県クイズの結果を推薦シグナルに含めること（B003 と統合後に検討）。
- 推薦履歴を DB に永続化して長期分析すること。
- 推薦比率（復習 vs 新規）や目標正答率帯のユーザー設定。
- 機械学習モデルを用いた推薦。
- 通知（プッシュ・メール）による「おすすめクイズ」リマインド。
