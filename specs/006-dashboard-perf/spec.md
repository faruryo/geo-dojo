# Feature Specification: ダッシュボード（トップ）表示速度の改善

**Feature Branch**: `006-dashboard-perf`
**Created**: 2026-06-08
**Status**: Draft
**Type**: パフォーマンス改修（新規ユーザー機能なし・既存挙動を維持したまま高速化）

> 注: 本件は新機能ではなく性能改修のため、軽量 spec とする。詳細は `plan.md` / `tasks.md` を参照。

## Clarifications

### Session 2026-06-08

- Q: 初回表示時、ダッシュボードのフィルタ選択（期間・地域）はリロード間で保持されるか？ → A: 保持されない（常に既定 all/全国 で開始）。サーバは既定フィルタをプリフェッチすれば初回描画は正しい。
- Q: AC4 の数値一致リグレッションは自動テストで保証するか目視で済ませるか？ → A: 自動テスト必須。純粋化した read クエリ群に Vitest を追加し、固定シードで主要指標の一致を保証する。

## 背景・問題 (Problem)

トップページ `app/(app)/page.tsx`（認証後ダッシュボード）の初回表示が遅い。本番 HAR 実測で **安定表示まで約14.3秒**。

### 実測エビデンス（本番 HAR: geo-dojo.faru.jp.har, 2026-06-08）

- トップ表示で **11本の Server Action（POST `/`）** が発火。
- **並列実行ゼロ（重なり 0/10 ペア）** — 各リクエストは前のが終わった約3ms後に開始。完全な直列。
- 総時間 ≒ 全アクションの単純合計 ＝ **14.3秒**。
- 各リクエストの所要時間の **約99%が `wait`（サーバ処理＝認証＋DB）**。受信は1〜2msで、転送量・バンドルは無罪。

| 主因 | 内容 |
|------|------|
| ① Server Action 直列化 | Next.js は client から呼ぶ Server Action を1本ずつ実行。react-query が並列発火しても直列になる。`(app)/layout.tsx` は `force-dynamic` でキャッシュも効かない |
| ② 認証の往復過多 | 各アクションが毎回 `supabase.auth.getUser()`（Auth サーバへネットワーク往復）。レイアウト分含め10回超が直列に積層 |
| ③ 個別クエリが重い | `getDashboardSummary` は `Promise.all` なしで直列10クエリ。`getRecommendation` は `municipality_master`（約1900行）＋全 quiz_results をロードして推薦エンジンを毎回実行 |
| ④ 重複フェッチ | `getRecommendation` が2回呼ばれている（2.5s＋1.0s＝3.5秒のムダ）。`RecommendHeroCard` が `page.tsx` の2箇所に記述、かつ `staleTime:0, refetchOnMount:'always'` |

## ゴール (Goals)

- 初回トップ表示（データ揃うまで）を **3秒未満**、可能なら **1秒台** に短縮する。
- 既存のダッシュボード表示内容・数値・フィルタ挙動を変えない（リグレッションなし）。
- DB インデックスは現状で適切なため、クエリチューニングではなく **取得アーキテクチャの並列化** を主眼とする。

## 非ゴール (Non-Goals)

- ダッシュボードの UI/UX や指標ロジックの変更。
- 新規テーブル・スキーマ変更。
- SRS / 推薦アルゴリズム自体のロジック変更。
- フィルタ選択の永続化（リロードで既定に戻る現行挙動を維持。サーバプリフェッチは既定 all/全国 を取得する）。

## 受け入れ基準 (Acceptance Criteria)

- [X] AC1: トップ初回表示でサーバ往復が並列化され、ウォール時間が単純合計ではなく最遅1本ぶんに収束する。→ **preview HAR 実測で達成**: 初回 read Server Action 11→1、サーバ側 9 read が並列 **1.6s** に収束（直列合計 ~14.3s）。
- [X] AC2: 同一データの重複フェッチ（特に `getRecommendation`）が解消されている。→ US1 で解消（単一マウント＋staleTime でキャッシュ共有）。
- [~] AC3: 認証検証の往復が初回表示あたり大幅に削減される。→ **実装完了**（プリフェッチで認証1回＋`getClaims` ローカル検証）。実効往復ゼロは本番の非対称署名鍵有効化が前提（research.md チェックリスト）。
- [X] AC4: 表示される全指標・グラフ・フィルタ結果が改修前と一致する。→ 純粋化 read に Vitest 必須（DBあり 50 passed）。SQL・集計・返却 shape 不変。推薦提示内容の画面目視はログイン環境で最終確認。
- [X] AC5: 初回表示が **< 3秒**。→ preview 実測でダッシュボード read は認証~0.8s＋並列~1.6s で揃う（< 3秒達成）。推薦カードのみ client 取得（loading 表示、~1.5s）。

## 検証方法 (Verification)

- 改修前後の HAR を取得し、(a) 直列/並列の重なり、(b) 総ウォール時間、(c) リクエスト本数を比較。
- 主要数値（totalQuestions / coverageRate / streak / weakness / 復習件数）の改修前後一致は、固定シードに対する純粋関数 Vitest（必須）で保証する。目視はあくまで補助とする。
