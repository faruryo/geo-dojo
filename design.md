# GeoDojo 設計書

## Context

GeoGuessr（特に日本）で勝つには、電柱・ガードレール・標識・植生といった地域固有の視覚特徴を
「瞬時に」判断する必要がある。これは知識の有無より「記憶の引き出し速度」の問題。
闇雲にゲームを繰り返すより、記憶科学に基づいた体系的な学習で習得を高速化するWebアプリを設計する。

**記憶理論ベース**：「科学的根拠に基づく最高の勉強法」（安川康介）
→ アクティブリコール・分散学習（SRS）・インターリービングの3本柱

---

## 機能候補 20案

各機能にユーザーストーリーを書き、最終的に3-4個に絞る。

---

### F01. アノテーション付きフラッシュカード
**概要**: 画像の「注目すべき部分」に丸や矢印でアノテーションを付けてカード化。
学習時はアノテーションを表示→隠す→答えるの順で進む。

**ユーザーストーリー**:
> 電柱の写真を見たとき、どこを見るべきかわからない。アノテーション付きカードで
> 「ここを見ろ」を学んでから、次回は自分で気づけるようになりたい。

**評価**: 記憶効果◎ / GeoGuessr直結◎ / 差別化◎（Ankiにない） / 開発コスト中

---

### F02. 4択地域クイズ
**概要**: 画像を見て4つの都道府県から正解を選ぶ。
選択肢は似た地域（隣県・同地方）を混ぜてひっかけを作る。

**ユーザーストーリー**:
> GeoGuessrで「青森か秋田かわからない」という状況を再現したい。
> 答えだけでなく「なぜ青森と間違えたか」の解説が見たい。

**評価**: 記憶効果○ / GeoGuessr直結◎ / 差別化△ / 開発コスト低

---

### F03. 地図タップクイズ
**概要**: 日本地図を表示し、画像の場所だと思う都道府県をクリックして回答。
GeoGuessrの操作感をそのまま再現する。

**ユーザーストーリー**:
> 選択肢があると「除去法」で答えてしまい、本当の記憶にならない。
> 地図を見て自力でピンを刺す練習がしたい。

**評価**: 記憶効果◎ / GeoGuessr直結◎◎ / 差別化◎ / 開発コスト中

---

### F04. 比較学習（サイドバイサイド）
**概要**: 似た地域の特徴画像を左右に並べて違いを体感する学習モード。
例：「東北3県の電柱を並べて、どこが違うか見つける」

**ユーザーストーリー**:
> 東北の県はいつも混乱する。「岩手vs秋田」を並べて
> 確実に違いを覚えたい。個別カードでは気づけない差異がある。

**評価**: 記憶効果◎ / GeoGuessr直結◎ / 差別化◎◎ / 開発コスト中

---

### F05. スピードランモード
**概要**: 30秒以内に1問→即判定を繰り返す。思考なく直感で答える訓練。
GeoGuessr本番では5秒以内に判断することもある。

**ユーザーストーリー**:
> 知識はあるのに本番で時間が足りない。「考えなくても反射で出る」
> レベルまで身につけたい。

**評価**: 記憶効果○ / GeoGuessr直結◎ / 差別化○ / 開発コスト低

---

### F06. 実戦スクリーンショット→即カード化
**概要**: GeoGuessrのプレイ後、スクリーンショットをアップロードすると
「場所・カテゴリ・気づいた特徴」を入力してカード化できる。
実戦と学習のループを最短化する。

**ユーザーストーリー**:
> プレイ中「これ何だろう」と思ったシーンを、ゲームを中断せずに
> メモしておきたい。後で見返してカードにしたい。

**評価**: 記憶効果◎（文脈記憶） / GeoGuessr直結◎◎ / 差別化◎ / 開発コスト低

### F07. 地域ガイドページ（読み物）
**概要**: 都道府県ごとの「GeoGuessr攻略特徴まとめ」ページ。
電力会社マップ、ガードレール色分布、特有の植生などを整理した参照資料。

**ユーザーストーリー**:
> ある県を集中対策したい。その県の特徴を体系的にまとめた
> ページを読んでから問題に臨みたい。

**評価**: 記憶効果△（受動的） / GeoGuessr直結◎ / 差別化○ / 開発コスト高（コンテンツ）

### F08. デイリーチャレンジ
**概要**: 毎日同じ5問が全ユーザーに出題される。
正答率・回答速度のランキングがある（将来的に）。

**ユーザーストーリー**:
> 毎日続けるための「今日のノルマ」が欲しい。
> 5分で終わる量にして学習習慣を作りたい。

**評価**: 記憶効果△ / GeoGuessr直結○ / 差別化△ / 開発コスト中

### F09. SRS復習キュー（スケジュール管理）
**概要**: SM-2アルゴリズムで計算した「今日復習すべきカード」を管理。
起動すると「今日やること」が一目でわかる。

**ユーザーストーリー**:
> 覚えたつもりのカードを1週間後に忘れてしまう。
> 最適なタイミングで自動的に復習が来てほしい。

**評価**: 記憶効果◎◎ / GeoGuessr直結○ / 差別化○（Ankiと同等） / 開発コスト中

### F10. 弱点集中モード
**概要**: 過去の回答履歴から「正答率が低いカード」だけを集中出題。
「知ってる問題」を飛ばして「苦手なもの」だけに時間を使う。

**ユーザーストーリー**:
> 東北の問題でいつもつまずく。「東北の電柱・苦手カードだけ30分集中」

**評価**: 記憶効果◎ / GeoGuessr直結◎ / 差別化○ / 開発コスト低

### F11. セッション後サマリー（White Paper法）
**概要**: 学習セッション終了後、「今日見たカードの特徴一覧」を表示。

**ユーザーストーリー**:
> フラッシュカードを終えても「本当に覚えたか」自信がない。
> 終わった後に「これだけ覚えた」という手ごたえを感じたい。

**評価**: 記憶効果◎（白紙勉強法） / GeoGuessr直結○ / 差別化◎ / 開発コスト低

## 実装対象（MVP）：4機能

記憶効果・GeoGuessr直結度・差別化の3軸で最も高い組み合わせを選ぶ。

---

### 🥇 F09. SRS復習キュー ＋ F01. アノテーション付きフラッシュカード（セット）
> **体験の核心**：「今日何を復習すべきか」が起動した瞬間にわかり、
> 画像の「どこを見るか」が明確なカードで答えを引き出す。

**学習フロー**：
```
アプリを開く
→ 「今日の復習: 12枚 / 新規: 5枚」が表示される
→ カードが1枚表示（電柱の写真＋丸アノテーション「←ここを見て」）
→ 地域を答える
→ 自己評価（1-5段階）→ 次回日程が自動設定
→ 12枚完了 → セッションサマリー表示
```

---

### 🥈 F03. 地図タップクイズ
> **体験の核心**：選択肢を見て「除去法」ではなく、日本地図を見て
> 「ここだ」と感じる直感を鍛える。GeoGuessr本番の操作感に最も近い。

**学習フロー**：
```
画像が表示される
→ 日本地図が表示される（都道府県の輪郭）
→ 「ここ！」と思う県をクリック
→ 正解位置と自分の回答が地図上に表示
→ 「正解は岩手県。秋田との違いはXX」と解説
```

---

### 🥉 F06. 実戦スクリーンショット→即カード化
> **体験の核心**：GeoGuessrで「これ何だろう」という体験を
> 学習に直結させる。実戦→記録→復習のループを作る。

**学習フロー**：
```
GeoGuessrをプレイ中に気になった場所でスクリーンショット
→ アプリに戻り「+追加」ボタン
→ スクリーンショットをアップロード
→ 都道府県・カテゴリ・気づいた特徴を入力
→ カードとして保存 → SRSキューに自動追加
→ 翌日から復習対象になる
```

---

### 🏅 F04. 比較学習（サイドバイサイド）
> **体験の核心**：「似ている地域を混同する」という最も多い失敗パターンへの直接対策。
> 「東北各県の電柱」を並べて「岩手だけ木製が多い」を体感で覚える。

**学習フロー**：
```
「比較学習」モードを選択
→ 混乱しやすいペアを選ぶ（例：岩手 vs 秋田）
→ 左：岩手の電柱写真、右：秋田の電柱写真
→ 下部に「相違点ハイライト」が表示される
→ 「さあ問題」→片方の画像で地域を答える
```

## スモールスタート：1〜2本から始める

### 実現できるユーザーストーリー（機能別）

| スコープ | 機能 | ユーザーストーリー |
|---|---|---|
| 最小 | F06のみ | GeoGuessrで気になった場所のスクショを撮って、後で見返せるようにメモしたい |
| 小 | F01（SRSなし） | 電柱の写真を見て「どこの都道府県か」を自分で答えて、正解を確認したい |
| 中 | F01 + F09（SRS付き） | 最適なタイミングで復習が来て、忘却曲線に沿って効率的に覚えたい |
| AI自動 | AI生成パイプライン | 毎日自動でカード候補が増えて、承認するだけでデッキが育っていく体験をしたい |

### 推奨スタート：案B（F06 + F01簡略版）

**案A（最も早く動くもの）: F06「スクショ→カード保存」のみ**
- できること：写真アップロード → 都道府県・カテゴリ入力 → カード保存
- できないこと：学習・復習機能なし（一覧を眺めるだけ）
- 工数：3〜5時間

**案B（推奨）: F06「カード作成」+ F01簡略版「フラッシュカード表示」**
- できること：カード作成 → ランダム表示 → 答えを確認（SRSなし）
- SRSは後から追加。まず「見て答える」体験を完成させる
- 意義：データ収集と基本的な学習サイクルが両方揃う
- 工数：6〜10時間

AI自動生成パイプラインは案Bの後に追加する。

## データ収集方法（優先度順）

### 最低必要枚数の考え方
- SRSが動くには全カード最低30枚（枚数不足だと「今日やること0枚」が続く）
- Phase 1目標：主要地方×各3〜5枚 = **30〜50枚**でMVP検証
- 理想：47都道府県×5カテゴリ×3枚 = 705枚（Phase 3以降）

### カテゴリ別特性

| カテゴリ | 地域差の明確さ | 取得しやすさ |
|---|---|---|
| 電柱（木製・コンクリート） | 高 | 中 |
| ガードレール（色・形状） | 高 | 中 |
| 標識（色・フォント） | 中 | 中 |
| 植生（樹種・農地パターン） | 中 | 高 |
| 建物・屋根 | 中 | 要注意（個人宅） |

---

### 優先度1：既存コミュニティリソースの活用（最速・最多）

GeoGuessr攻略コミュニティはすでに大量の整理済みデータを無償公開している。
これを加工・構造化するだけで初期カードを大量に作れる。**新規データ収集より先にここを使い切る。**

**主要リソース一覧**:

| リソース | URL | 内容 | 活用方法 |
|---|---|---|---|
| **Plonkit** | plonkit.net | 国・地域別の識別ヒント集（画像＋テキスト）。日本の都道府県別にも整備済み | 画像＋特徴テキストをそのままカード化 |
| **GeoTips.net** | geotips.net | タグ検索可能な地域別ヒント集 | 特徴テキストの参照・引用元として |
| **Reddit r/geoguessr** | reddit.com/r/geoguessr | 攻略Tips投稿・Wikiまとめ（日本解説あり） | 特徴テキストの補完・アノテーションのヒント参照 |
| **YouTube攻略動画** | - | 「Japan GeoGuessr tips」で多数。画像＋解説付き | スクリーンショット＋解説テキストをカード化 |
| **日本語Discordサーバー** | - | GeoGuessr日本コミュニティが集めた都道府県別まとめ画像 | 画像とテキストをそのまま活用 |

**Plonkit の具体的な活用手順**:
```
1. plonkit.net/japan にアクセス → 都道府県別ヒントページを確認
2. 各ページの「識別画像」をスクリーンショットまたはダウンロード
3. 特徴テキストを自分の言葉に言い換えてカードに登録
4. 画像が使えない場合は「特徴テキストのみカード」として登録し、
   後から Street View APIで対応画像を補完する
```

**著作権上の扱い（重要）**:
- Plonkitなどの画像の多くは元をたどればGoogle Street Viewのスクリーンショット。厳密な再配布権はない
- **アプリ側がPlonkitをスクレイピングしてDBに初期データとして組み込むことは禁止**（二次著作権侵害リスク）
- 推奨運用：「空のプラットフォーム」として提供し、ユーザー自身がPlonkitを参照しながら手動でカードを作成するUGCモデルを採用
  - テキスト情報（特徴の説明）は自分の言葉に言い換えて手入力
  - 画像は Street View API 経由で動的取得（pano_idを保存）する方式に統一

**期待できる初期データ量**:
- Plonkit Japan：47都道府県 × 3〜8枚程度 ＝ **150〜300枚**が数時間で確保可能
- これだけでSRSが十分機能する枚数（30枚の壁をはるかに超える）

### 優先度2：手動キュレーション（Google Street View）
優先度1で不足している地域やカテゴリを補完する用途。Google Street Viewで訪問→スクリーンショット→Canva/Figmaでアノテーション付与。

- 優先度1の穴埋め（Plonkitにない地域・カテゴリ）に集中する
- **著作権注意**：スクリーンショットをアプリに蓄積・再配布するのはグレーゾーン → 優先度3と組み合わせる

### 優先度3：Google Street View Static API による自動取得
緯度経度・カメラ角度のJSONリストを手作業で作成 → スクリプトでAPIバッチ取得。

- コスト：月$200の無料クレジット内（10枚/日×30日=$2.1）
- **著作権対応**：「APIキー経由で動的フェッチして表示」はOK。画像をDBに蓄積・再配布はNG

### 優先度4：Mapillary API（オープンデータ）
MetaのCC BY-SAライセンス街路写真。バウンディングボックスで地域フィルタリング可能。
日本の地方部はカバレッジが薄いため手動フィルタリングが必要。

### 優先度5：AI自動生成パイプライン（毎日データが増える仕組み）

Street View Static APIとAIを組み合わせて、**毎日自動でカード候補を生成**するバックグラウンドパイプライン。

```
① Cron（毎日1回） → 代表地点リストからランダムにピックアップ
② Google Street View Static APIで画像取得
③ Gemini 2.5 Flash にマルチモーダルで送信：
   - カテゴリ判定（電柱/ガードレール/標識/植生/判定不可）
   - 特徴テキスト生成（100字以内）
   - アノテーション座標の提案（x_pct, y_pct）
④ 信頼度0.7以上のものを「draft」状態でDBに保存
⑤ 承認UIでユーザーが確認→承認されたらSRSキューに追加
```

**AIへのシステムプロンプト例**:
```
この画像はGoogle Street Viewの日本の街路写真です。
GeoGuessr上達のためのフラッシュカード学習アプリ用に分析してください。

出力JSON:
{
  "category": "denchu|guardrail|sign|vegetation|building|other|skip",
  "confidence": 0.0-1.0,
  "feature_text": "（この画像の地域識別に使える特徴を100字以内で）",
  "annotation_hints": [{"x_pct": 0.45, "y_pct": 0.3, "label": "木製電柱"}]
}

confidence < 0.7 の場合は category を "skip" にしてください。
```

```text
                      └──────┬──────────┬────────────┘

                    ┌────────┘          └────────────┐
                    ▼                               ▼
          ┌──────────────┐              ┌──────────────────┐
          │ annotations  │              │   srs_records    │
          │──────────────│              │──────────────────│
          │ id           │              │ id               │

          │  circle/arrow│              │ ease_factor      │
          │ x_pct        │              │ interval_days    │

          └──────────────┘              └──────────────────┘

```

### フィールド詳細

**srs_records（SM-2アルゴリズム）**:
- `ease_factor`：初期2.5、最小1.3
- `interval_days`：初回=1日、2回目=6日、以降=前回×ease_factor
- `repetitions`：連続正解回数（0にリセットで再学習）

**今日の復習キュー取得**:
```sql
-- パフォーマンス最適化インデックス（必須）
CREATE INDEX idx_srs_due_date_user ON srs_records (user_id, due_date);

-- 期限が来た復習カード
SELECT c.*, s.ease_factor, s.due_date
FROM cards c
JOIN srs_records s ON s.card_id = c.id
WHERE s.user_id = :user_id AND s.due_date <= CURRENT_DATE
ORDER BY s.due_date ASC LIMIT 20;

-- 新規カード（SRSレコードなし）
SELECT c.* FROM cards c
LEFT JOIN srs_records s ON s.card_id = c.id AND s.user_id = :user_id
WHERE s.id IS NULL AND c.user_id = :user_id
LIMIT 5;
```

- Tailwind：モバイルファースト設計と相性が良い。ダークモードも容易
- shadcn/ui：Button・Card・Sheet・Select・Dialog など、必要なコンポーネントを必要な分だけ追加
- アニメーション：カードフリップは CSS `perspective + rotateY`（Framer Motion は不要）

#### バックエンド・DB：Supabase

| 機能 | 使い方 |

| Supabase Storage | アップロード画像（Free tier 1GB） |
| Edge Functions | AI自動生成Cronジョブ（Deno Cron） |
| Row Level Security | `user_id` ベースのアクセス制御 |

- ORM：**Drizzle ORM**（TypeScript-first・軽量・Supabase との相性◎）
  - `schema.ts` でテーブル定義 → `drizzle-kit` でマイグレーション
  - Prisma より起動が速くエッジ対応

#### データ取得：TanStack Query（React Query v5）

- Server Actions（カード保存・自己評価登録）との役割分担を明確にする
  - 読み取り（重い） → TanStack Query
  - 書き込み → Server Actions

#### 地図コンポーネント：@vnedyalk0v/react19-simple-maps

- **`react-simple-maps`（オリジナル）は使用禁止**：React 18止まり、2026年環境で動作不安定
- React 19対応フォーク `@vnedyalk0v/react19-simple-maps` を使用（ESM専用・TypeScriptファースト）
- 地図データは GeoJSON ではなく **TopoJSON** 形式に変換して使う
  - `mapshaper` でビルド前に変換：GeoJSON → TopoJSON（Visvalingamアルゴリズムで単純化）
  - 変換後ファイルは `public/japan-topo.json` に配置（約1.3MB以下を目標）
  - コンポーネントのマウント時に非同期フェッチ（バンドルに含めない）
  - Serwistのランタイムキャッシュで2回目以降はネットワーク不要
- 各 `<path>` にクリックイベント → 都道府県選択
- 国土地理院GeoJSON（CC BY 4.0）を変換元に使用

#### アノテーション：SVG オーバーレイ（ライブラリなし）

- 画像の上に `<svg>` を `position:absolute` で重ねる
- `<circle>` と `<line>` で丸・矢印を描画
- 座標は画像の相対パーセント（0.0〜1.0）で保存
- 編集UI（F06）：クリック位置 → `clientX/Y` を画像サイズで割って保存
- react-konva や fabricjs は不要（シンプルな丸・矢印のみ）

#### AI自動生成：Gemini 2.5 Flash

> ⚠️ Gemini 2.0 Flash は **2026年6月1日でシャットダウン予定**。Gemini 2.5 Flash を使うこと。

```
Supabase Edge Function（Deno Cron：毎日AM4時）
  └─ 代表地点JSONからランダム10地点をピックアップ
  └─ Google Street View Static APIで画像URL生成
  └─ Gemini 2.5 Flash（multimodal）に画像を送信
  └─ JSON応答（category, confidence, feature_text, annotation_hints）
  └─ confidence ≥ 0.7 のものを cards テーブルに source='ai_generated', visibility='draft' で挿入
```

- SDK：`@google/generative-ai`（Deno対応）
- 承認UIは Next.js の管理ページ（`/admin/review`）

#### 画像ストレージ・Street View 動的レンダリング

**Street View 画像の扱い（利用規約厳守）**:

| データ要素 | 保存先 | 保存可否 |
|---|---|---|
| パノラマ画像本体 | ❌ 保存禁止 | セッション中のみブラウザメモリ |
| pano_id（パノラマID） | ✅ Supabase DB | 無期限保存OK（規約上の例外） |
| アノテーション座標（x_pct, y_pct） | ✅ Supabase DB | ユーザー生成メタデータとして保存可 |
| AI抽出の特徴テキスト | ✅ Supabase DB | 保存可（画像派生物の解釈に注意） |

- **Street View画像をSupabase Storageに保存することは規約違反（第3.2.3条）**
- 表示時の流れ：DB から `pano_id` を取得 → `/api/image-proxy` 経由でGoogleに動的フェッチ → クライアントが `<img>` に表示
- `/api/image-proxy` Route Handler でAPIキーを隠蔽（クライアントに露出させない）
- ブラウザキャッシュ（`Cache-Control`）+ Serwistランタイムキャッシュで重複リクエストを削減

**ユーザーアップロード画像（F06スクリーンショット）**:
- 私的学習目的の範囲内で Supabase Storage に一時保存を許容
- アップロード前にブラウザ側でリサイズ（`browser-image-compression`）：最大1080px・品質80%・JPEG
- `cards/{user_id}/{card_id}.jpg` のパスで保存

#### PWA：@serwist/next

> ⚠️ `@ducanh2912/next-pwa` は**使用禁止**：Next.js 15 + Turbopack環境で動作しない（2年以上更新停止）

```
@serwist/next（Workboxフォーク）の採用：

```

体験：スマホにインストール → GeoGuessrと2画面並行利用 → オフラインでも既存カードを復習可能

### ❌ やってはいけないこと

| # | 禁止事項 | 理由 |
|---|---|---|
| 1 | `@ducanh2912/next-pwa` を使う | Next.js 15 + Turbopack環境で動作不可（2年以上更新停止） |

| 5 | AIパイプラインで取得した画像をDBに永続化する | 同上。画像は破棄し、pano_idとメタデータのみ保存する |

| 9 | APIキーにクォータ上限を設定せず運用する | 月10,000req超で$7/1,000reqの課金が突然発生するリスク |
| 10 | Street View APIキーをクライアントに露出させる | 不正利用・コスト爆発リスク。`/api/image-proxy` 経由必須 |

### ✅ やること・守ること

| # | 要件 | 詳細 |
|---|---|---|
| 1 | Next.js は **15.2.6以降**を指定 | `package.json` に `"next": ">=15.2.6"` でバージョンを固定 |
| 2 | PWAは `@serwist/next` を使う | `withSerwist` ラッパー + `public/sw.js` 生成 |
| 3 | 地図は `@vnedyalk0v/react19-simple-maps` + TopoJSON | `mapshaper` でGeoJSON→TopoJSON変換してビルドパイプラインに組み込む |
| 4 | Street Viewは動的レンダリング | pano_idをDBに保存し、表示時に `/api/image-proxy` 経由で動的フェッチ |
| 5 | アノテーション座標は相対値（0.0〜1.0）で保存 | 画像サイズに依存しないレスポンシブ対応のため |
| 6 | AIパイプラインは必ずHITLを挟む | AI生成カードはdraftとして保存 → 人間が承認後にSRSキューへ |
| 7 | Gemini 2.5 Flash を使う | 入力$0.30/1M tokens、出力$2.50/1M tokens。月約¥30 |
| 8 | SRSテーブルに複合インデックスを設定 | `CREATE INDEX idx_srs_due_date_user ON srs_records (user_id, due_date)` |
| 9 | Google Maps APIキーに日次ハードリミットを設定 | 330 req/日を上限にして月10,000req無料枠を超えないよう物理遮断 |
| 10 | Plonkitはテキスト情報の参照のみ | 画像は使わず、特徴テキストを自分の言葉で手入力してUGCモデルで運用 |
| 11 | コンテンツ投稿はUGCモデルで運用 | プラットフォーム側はカードの入れ物を提供。コンテンツはユーザー自身が作成 |
| 12 | SM-2評価は3段階に簡略化 | 全然(1) / うろ覚え(3) / 完璧(5)。モバイルで5段階は煩雑 |
